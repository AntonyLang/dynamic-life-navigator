# Product Requirements Document (PRD) V2: Dynamic Life Navigator

## 1. 产品定位
Dynamic Life Navigator 是一个面向“灵活型、自驱型、任务与兴趣并行”的个人导航系统。
它不以“替用户排满时间表”为目标，而以“在当前状态下给出最合适的下一步”为目标。

系统通过整合：
- 用户主动输入（聊天、语音、转发内容）
- 外部被动事件（运动、代码、日历等）
- 结构化行动节点库（项目/价值流）
- 实时富集信息（天气、文章、路线、播客等）

在 Pull（用户主动问）和有限 Push（系统判断时机合适）两类场景下，生成低打扰、高相关、可立即执行的建议。

---

## 2. 核心用户与使用场景
### 2.1 目标用户
- 不喜欢刚性日程块，但并非没有目标
- 同时管理学习、项目、运动、兴趣等多条线
- 经常出现“有空但不知道现在做什么最合适”的状态
- 希望输入成本极低，不愿维护复杂清单

### 2.2 核心问题
现有 Todo / Calendar 工具的主要问题：
- 强调“计划完整性”，不关心用户当前状态
- 能记录任务，但不理解“刚经历了什么”
- 缺乏上下文与实时信息，无法给出真正可执行的建议
- Push 往往基于固定时间，而非时机

### 2.3 核心价值
系统要解决的不是“记住任务”，而是：
1. 当前状态下我适合做什么
2. 哪件事现在最值得做
3. 有没有更轻、更顺手的替代选项
4. 在不增加维护成本的前提下，让目标持续推进

---

## 3. 产品目标
### 3.1 MVP 目标（第一阶段）
MVP 只解决一个最小闭环：
- 单用户
- 两个交互入口：桌面插件 + 企业微信/微信机器人（二选一先落地）
- 一个主动输入通道（聊天）
- 一个被动事件源（推荐 Strava 或 GitHub，二选一）
- 一个 Pull 建议接口
- 一个低频 Push 机制
- 10~30 个活跃行动节点

### 3.2 非目标（MVP 不做）
- 多用户租户系统
- 复杂权限与团队协作
- 全量多模态理解（图片 OCR、视频理解等）
- 完整知识库/搜索引擎
- 自动代办拆解到极细粒度
- 高频、强运营式消息推送

### 3.3 成功指标
#### 北极星指标
- Pull 请求后的建议采纳率

#### 核心产品指标
- 7 日内至少一次有效交互的活跃天数
- 建议采纳率（accepted / shown）
- 建议忽略率与明确拒绝率
- Push 打扰率（dismissed / pushed）
- 单次推荐平均生成耗时
- 节点覆盖率（30 天内被命中过的活跃节点比例）

## 4. 核心设计原则
### 4.1 Zero-Friction Input
输入要足够轻：
- 可直接说人话
- 可发语音转文本
- 可转发内容给机器人
- 不要求用户先整理后输入

### 4.2 State-Aware, Not Schedule-Aware Only
系统首要理解的是“现在的状态”，不是“日历上的空格”。

### 4.3 Structured Retrieval First, LLM Rendering Second
先用结构化规则和分数快速收敛候选，再用 LLM 做最终表达与少量排序解释。

### 4.4 Push Should Be Rare and Earned
Push 必须克制。只有在明显存在价值、且打扰成本可接受时才推。

### 4.5 Every Recommendation Must Be Auditable
每次建议都必须能追踪：
- 为什么命中这几个节点
- 用了哪些状态
- 是否被采纳
- 是否造成打扰

---

## 5. 统一领域模型
### 5.1 用户状态（User State）
系统维护一个“当前状态快照”，至少包含：
- mental_energy：脑力值（0~100）
- physical_energy：体力值（0~100）
- focus_mode：当前偏好状态（如 focused / tired / bored / commuting）
- do_not_disturb_until：免打扰截止时间
- recent_context：最近一次重要事件摘要
- state_version：乐观锁版本号

### 5.2 行动节点（Action Node）
节点是建议的基本候选单元，分两类：
- project：有明确终点和 deadline
- value：无明确终点，但长期值得持续投入

节点最少应有：
- title / summary
- tags
- energy requirements
- 预计时长
- deadline（可空）
- 优先级与权重
- cooldown 规则
- 最后推荐时间 / 最后完成时间
- 当前状态（active / paused / archived / done）

### 5.3 事件（Event）
事件来自主动输入或外部系统。每个事件必须记录：
- 来源
- 原始载荷
- 解析结果
- 对状态的影响
- 是否已压缩归档
- 幂等键（如果来自 webhook）

### 5.4 推荐记录（Recommendation Record）
每次 Pull 或 Push 都要落库，记录：
- 推荐模式（pull / push）
- 候选节点列表
- 最终命中节点
- 触发原因
- 文案
- 是否展示成功
- 用户反馈（accepted / ignored / dismissed / rejected）

## 6. 功能范围
### 6.1 输入层
- 自然语言消息录入
- 语音转文本后录入
- 外部 webhook 接入
- 内容转发入库

### 6.2 状态层
- 根据事件更新用户状态
- 记录状态变化原因
- 支持重置与手动修正

### 6.3 推荐层
- Pull：用户主动问“现在做什么”
- Push：状态突变或 deadline 临近时触发评估
- 简报：返回当前活跃节点和推荐理由摘要

### 6.4 富集层
- 定时为活跃节点抓取补充信息
- 给节点增加短时情报，但必须带有效期

### 6.5 压缩层
- 定时汇总事件日志
- 更新长期标签与节点权重
- 归档低价值原始流水

---

## 7. 推荐治理规则
### 7.1 候选筛选硬规则
- 节点状态必须为 active
- mental / physical 要求不能高于当前状态
- 在 cooldown 期内的节点不推荐
- 被用户最近明确拒绝的节点短期降权或屏蔽
- 已过期的 live annotations 默认不参与文案拼装

### 7.2 排序参考维度
- deadline 紧迫度
- 当前状态匹配度
- 最近活跃度
- 长期价值权重
- 最近推荐频次惩罚
- 最近被拒绝惩罚
- 可执行时长匹配度

### 7.3 Push 触发条件
满足以下之一才允许进入 Push 评估：
- 明显状态突变（高强度任务后、运动后、长时间空档等）
- DDL 临界点到达阈值
- 长期价值节点超过最大未触达时间
- 新鲜情报命中且与当前状态高度匹配

### 7.4 Push 抑制条件
命中任一条件则不推：
- 在 do_not_disturb 期间
- 过去 N 小时已推过消息
- 当前没有高置信度候选
- 候选仅比随机建议略好，不值得打扰

## 8. MVP 用户旅程
### 8.1 主动输入 + Pull
1. 用户说“刚调完实验，脑子麻了”
2. 系统解析事件并扣减脑力值
3. 用户问“现在适合干嘛”
4. 系统快筛候选节点
5. 结合实时批注返回 1~2 条建议
6. 用户点击“采纳/略过/换一个”

### 8.2 被动事件 + Push
1. Strava/GitHub webhook 到达
2. 系统落库并更新状态
3. 触发 Push 评估
4. 命中高价值建议则发一条消息
5. 记录用户是否查看/忽略/采纳

---

## 9. 风险与约束
### 9.1 产品风险
- 过度 Push 导致用户反感
- 状态推断不准导致“懂你但不准”
- 节点维护不足导致候选池枯竭

### 9.2 技术风险
- webhook 幂等与安全校验不足
- LLM 输出不稳定导致解析失败
- 外部 API 质量与可用性波动

### 9.3 设计约束
- 所有自动推断都允许被用户纠正
- 高风险场景不做强结论，不臆造实时事实
- 文案可以拟人，但推荐逻辑必须可追踪

---

## 10. 发布建议
### Phase 1：最小闭环
- 单入口
- 单 webhook
- Pull 为主，Push 为辅
- 手工维护节点库

### Phase 2：推荐治理完善
- 增加反馈回流
- 增加简报与节点复盘
- 增加 annotation 有效期与冷却策略

### Phase 3：扩展接入
- 增加更多外部源
- 增加更多客户端
- 提升个性化排序能力
