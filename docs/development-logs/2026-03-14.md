## Step 36: replay / rebuild correctness hardening
- Audited the new replay/rebuild operator tooling and confirmed two real correctness issues:
  - bounded historical rebuilds were being compared against the current persisted `user_state`, creating false drift
  - replay rows that shared a timestamp with the checkpoint anchor could be skipped because the range cut only used `created_at > anchor_created_at`
- Updated `app/services/replay_service.py` so rebuild now:
  - uses the same `(created_at, id)` cursor for both ordering and anchor cut
  - exposes an explicit `comparison_target`
  - compares bounded windows against:
    - the last replayed `state_history.after_state`, or
    - the anchor snapshot when no replay rows exist
  - keeps comparison against current persisted state only for unbounded "rebuild to now"
- Tightened rebuild event stats so the bounded-window query now aligns with the authoritative anchor/time range rather than a looser current-state approximation.
- Extended replay coverage in `tests/test_replay_service.py` for:
  - bounded historical rebuilds
  - anchor + first patch sharing the same timestamp
  - bounded windows with no replay rows
  - latest event-patch selection for single-event replay
  - event stats at the anchor timestamp

## Review notes
- This was a correctness hardening step, not a product-surface change.
- The key goal was to restore trust in the operator dry-run tools before any future `rebuild --apply` discussion.
- `reduce_state_snapshot()` remains the single reducer source for both live writes and dry-run replay.
